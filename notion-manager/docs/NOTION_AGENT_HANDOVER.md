# 노션 관리 에이전트 (notion_manager) 개발 핸드오버 문서

> 이 문서는 Claude Code 기반 자율 에이전트가 **디스코드 봇 `notion_manager`** 를 개발하기 위한
> 전체 컨텍스트를 담고 있다. 대상 저장소: `https://github.com/Chohyungun/uipa-code-lab`
> 작업 브랜치: `feature/notion-manager-agent` (새로 생성할 것)

---

## 1. 프로젝트 목적

### 배경
- 사내 개발 스터디그룹(5명)이 Notion 워크스페이스로 학습/일정/자료를 관리한다.
- 팀의 소통은 전부 **Discord**에서 이루어진다. 회의 일정, 앞으로 할 일, 개인 공부에 필요한 것 등이
  디스코드 대화에서 나오는데, 이를 누군가가 일일이 노션에 옮기는 것이 비효율적이다.

### 목표
- **별도의 노션 관리자 없이**, 디스코드 봇 + Claude 자율 에이전트가 노션 관리를 대행한다.
- 사용자가 디스코드에서 봇에게 자연어로 요청 →
  Claude 자율 에이전트(**런타임 모델: Claude Opus 4.8**)가 Notion API로 처리 →
  처리 결과를 디스코드로 보고 → 사용자 피드백 반영 (수정/재시도).
- 참고: 이 봇을 **개발하는** Claude Code 에이전트는 fable-5로 구동되지만, 완성된 봇이
  **런타임에 호출하는** 에이전트 모델은 Opus 4.8이다. 두 역할을 혼동하지 말 것.

### 동작 흐름 (요구사항)
```
[Discord 사용자 메시지]
      │  (봇 멘션 또는 지정 채널 모니터링)
      ▼
[notion_manager Discord Bot]
      │  요청 파싱 → 에이전트 세션 생성/재사용
      ▼
[Claude 자율 에이전트 (Opus 4.8)]
      │  Notion API 호출 (아래 5장의 워크스페이스 맵 참조)
      ▼
[결과 보고: 디스코드 메시지/스레드로 요약 + 노션 링크]
      │
      ▼
[사용자 피드백 (예: "날짜 틀렸어, 다음주 수요일로")] ──▶ 같은 스레드에서 컨텍스트 유지하며 재처리
```

### 대표 사용 시나리오
| 디스코드 발화 예시 | 에이전트가 할 일 |
|---|---|
| "다음 모임 7/23 수요일 저녁 7시로 잡아줘" | 스터디 일정 DB에 항목 생성/수정 |
| "나 이번주에 랭체인 RAG 챕터 끝냈어" | 해당 멤버의 학습 트래커 항목 상태/메모 업데이트 |
| "이 강의 좋더라 (링크)" | 공유 자료실 DB에 자료 등록 (분야/난이도 추정) |
| "오늘 회의록 정리해줘: (요약 내용)" | 해당 회차 일정 페이지에 회의록 작성, 상태를 완료로 변경 |
| "박보배 다음주 발표로 바꿔줘" | 일정 DB의 발표/진행 필드 수정 |
| "이번 분기 김다영 목표에 ~ 추가해줘" | 해당 멤버 페이지의 분기 목표 체크리스트에 항목 추가 |

## 2. 기술 스택 및 아키텍처 가이드 (제안)

구현 언어/프레임워크는 에이전트가 판단하되, 아래를 권장한다.

- **Discord 봇**: Python `discord.py` 또는 Node `discord.js`
  - 필수: Discord Developer Portal에서 **MESSAGE CONTENT INTENT** 활성화 (자연어 요청을 읽어야 함)
  - 봇 권한: Read Messages/View Channels, Send Messages, Create Public Threads, Read Message History, Add Reactions
  - 트리거 방식: ① 봇 멘션(@notion_manager) ② 지정 채널(`#노션-관리`)의 모든 메시지 — 둘 다 지원 권장
- **에이전트 실행**: Claude Agent SDK 또는 Claude Code headless 모드 (`claude -p`)
  - 모델: **Claude Opus 4.8** — `.env`의 `CLAUDE_MODEL` 값 사용.
    정확한 API 모델 ID 문자열(예: `claude-opus-4-8-YYYYMMDD` 형태)은 구현 시
    https://docs.claude.com/en/docs/about-claude/models 에서 확인해 반영할 것
  - 요청 1건 = 에이전트 세션 1개. 피드백 루프를 위해 디스코드 **스레드 ID ↔ 세션 ID 매핑**을 저장해
    같은 스레드의 후속 메시지는 같은 세션으로 이어서 처리
- **Notion 연동**: Notion 공식 API (integration token 사용) 또는 Notion MCP 서버
  - 워크스페이스 맵(5장)의 ID들을 시스템 프롬프트/설정 파일로 에이전트에 주입할 것
- **안전장치 (필수)**:
  - 삭제/대량 변경 요청은 실행 전 디스코드에서 ✅ 리액션 확인을 받고 실행
  - 모든 변경은 처리 후 "무엇을 바꿨는지 + 노션 링크"를 보고
  - 에러 시 원문 에러를 숨기지 말고 요약해 보고, 재시도 여부를 물을 것

### 권장 저장소 구조
```

└── notion-manager/
    ├── README.md              # 이 문서 기반 요약

    ├── agent/                 # 에이전트 실행 래퍼, 시스템 프롬프트, 노션 워크스페이스 맵
    │   └── workspace_map.json # 5장의 ID들을 구조화한 설정 파일
```

## 6. 에이전트 운영 원칙 (시스템 프롬프트에 반영할 것)

1. **언어**: 모든 노션 콘텐츠와 디스코드 응답은 한국어. 팀은 비전공자 포함이므로 쉬운 표현 사용.
2. **구조 보존**: 기존 페이지 레이아웃/DB 스키마를 임의로 바꾸지 않는다. 스키마 변경은 사용자 확인 후.
3. **속성 규칙**: 트래커 항목 생성 시 `담당 멤버` 필수 지정. 일정 항목은 `상태=예정`으로 생성, 모임 후 `완료`로 갱신.
4. **회의록**: 회의록은 해당 회차 일정 DB 항목의 페이지 본문에 작성 (별도 페이지 생성 금지).
5. **모호한 요청**: 날짜·대상 멤버가 불명확하면 실행 전에 디스코드에서 되묻는다.
6. **보고 형식**: 처리 후 "✅ 한 일 요약 + 노션 링크" 형태로 응답. 실패 시 원인 요약 + 대안 제시.
7. **이름 주의**: 멤버 "김다영"의 트래커 select 옵션이 아직 "박다영"으로 남아 있음 — 최초 구동 시 rename 처리.
