# 2D 농장 시뮬레이션 게임 — 설계 문서

작성일: 2026-07-17
엔진: Godot 4.7.1 (GDScript)
위치: `Dayoung_Study/farm-game/` — 공용 레포이므로 모든 작업물은 `Dayoung_Study/` 하위에만 둔다

## 개요

스타듀밸리를 참고하되 핵심 루프 하나에 집중한 2D 농장 시뮬레이션 게임.
스터디 목적의 첫 Godot 프로젝트.

**핵심 루프**: 밭을 갈고 → 씨앗을 심고 → 물을 주고 → 하루가 지나면 자라고 → 수확하고 → 팔아서 → 다시 씨앗을 산다

## 결정 사항 요약

| 항목 | 결정 |
|---|---|
| 범위 | 핵심 루프 전체 (농사 + 체력 + 돈 + 상점). 마을/NPC/광산/낚시 없음 |
| 그래픽 | 색 도형 플레이스홀더. 스프라이트는 나중에 교체 |
| 이동 | 격자 기반, 한 칸씩 이동 |
| 시간 | 시계 + 체력 병행 |
| 하루 종료 | 시간 초과 또는 체력 0 시 기절 → 다음 날 체력 절반으로 시작 |
| 저장 | 처음부터 설계에 포함 |

## 1. 전체 구조

핵심 원칙은 **게임 상태와 화면 표시의 완전한 분리**다.

### FarmModel — 순수 데이터 + 규칙 (`RefCounted`, 노드 아님)

게임의 모든 사실과 규칙을 보유한다. 날짜, 시각, 돈, 체력, 밭 칸 상태, 인벤토리, 도구 레벨.

**이 객체는 Godot 노드가 아니며 렌더링에 대해 아무것도 모른다.** 순수한 데이터와 규칙만 존재한다.

이 설계에서 오는 이점:

- **테스트 용이성** — "3일차에 물 안 준 작물이 안 자라는가"를 확인하려고 게임을 3일 플레이할 필요가 없다. FarmModel을 생성하고 함수를 호출해 확인하면 된다. 씬 트리가 필요 없으므로 헤드리스로 실행 가능하다.
- **저장 단순화** — 저장은 이 객체를 JSON으로 직렬화하면 끝. 흩어진 상태를 그러모을 필요가 없다.

### GameState — autoload 껍데기 (`Node`)

**Godot의 autoload로 등록하는 스크립트는 `Node`를 상속해야 하며 씬 트리에 노드로 붙는다.** 따라서 순수 모델을 그대로 autoload할 수 없다. GameState는 `FarmModel` 인스턴스를 하나 들고 있는 얇은 껍데기 역할만 한다.

GameState가 담당하는 것 — 노드여야만 할 수 있는 일들:

- 시계 진행 (`_process`로 실제 시간 경과를 게임 시간으로 변환)
- 신호(signal) 발신 — 모델의 상태가 바뀌었음을 화면 노드들에게 알림
- 파일 입출력 — `user://save.json` 읽기/쓰기

**규칙 판정은 전부 FarmModel에 있다.** GameState는 호출을 전달하고 결과를 신호로 알릴 뿐, 스스로 판단하지 않는다.

### 화면 노드

GameState를 읽어 그리기만 한다.

| 노드 | 역할 |
|---|---|
| `Player` | 방향키 입력 → 격자 이동, 스페이스 → 앞 칸에 도구 사용 |
| `Farm` | 밭 칸 상태를 읽어 색 타일로 렌더링 |
| `HUD` | 시계, 체력 바, 돈, 선택된 도구 표시 |
| `Shop` | 씨앗 구매 / 작물 판매 UI |

### 데이터 흐름 (단방향)

```
플레이어 입력 (스페이스)
  → GameState.use_tool(위치, 방향, 도구)
  → FarmModel이 규칙에 따라 상태 변경
  → GameState가 신호 발신
  → Farm / HUD 가 신호를 듣고 화면 갱신
```

**화면 노드가 직접 상태를 변경하지 않는다.** 이 규칙 하나로 "화면과 실제 상태가 따로 노는" 버그가 구조적으로 차단된다.

## 2. 데이터 모델

### 게임 전역 상태 (FarmModel 보유, 저장 대상)

| 항목 | 설명 |
|---|---|
| `day` | 며칠차인지. 1부터 시작 |
| `hour`, `minute` | 현재 시각. 매일 아침 6:00으로 리셋 |
| `money` | 소지금 |
| `energy` / `max_energy` | 현재 체력 / 최대 체력 |
| `inventory` | 아이템 종류 → 개수. 씨앗과 수확물이 함께 들어감 |
| `tiles` | 밭 좌표 → 밭 칸 상태 |
| `tool_levels` | 도구 종류 → 레벨 (섹션 3 참고) |

### 밭 칸 (FarmTile)

| 항목 | 설명 |
|---|---|
| `state` | `UNTILLED` / `TILLED` / `PLANTED` |
| `crop_id` | 심어진 작물. 비어있으면 없음 |
| `growth` | 자란 일수 |
| `watered` | 오늘 물 줬는지 여부 |

### 작물 정의 (상수, 저장하지 않음)

고정 데이터이므로 코드에 상수로 둔다. 작물별 보유 항목: 이름, 성장 일수, 씨앗 가격, 판매 가격, 렌더링 색상.

3종으로 시작한다 — 빠르고 싼 것 / 중간 / 느리고 비싼 것.
"빨리 회전시킬까, 크게 걸까"라는 선택지를 만들기 위한 최소 구성.

### 성장 규칙

- 하루가 넘어갈 때, `PLANTED` 상태이고 `watered == true`인 칸만 `growth += 1`
- 물을 안 줬으면 그날은 자라지 않는다. **죽지는 않는다** (초반에 너무 가혹함)
- 매일 아침 모든 칸의 `watered`가 `false`로 리셋 → 매일 다시 물을 줘야 함. 이것이 하루 체력을 쓰는 주된 이유
- `growth >= 작물의 성장 일수`가 되면 수확 가능
- 수확 시 작물이 인벤토리로 이동, 해당 칸은 `TILLED`로 복귀 (다시 갈 필요 없이 바로 심기 가능)

### 명시적으로 제외한 것

- **계절 없음** — 계절별 작물/전멸 규칙은 재미보다 관리 부담이 먼저 온다
- 갈아놓은 밭의 시간 경과 원상복구 없음
- 잡초, 돌멩이 없음

### 구현 주의

밭 좌표를 키로 쓰는데 JSON은 좌표 타입을 키로 지원하지 않는다.
저장 시 `"3,5"` 형태의 문자열로 변환하고, 불러올 때 역변환한다.
저장 기능에서 가장 먼저 문제가 생기는 지점.

## 3. 상점과 도구 업그레이드

### 배경: 돈에는 쓸 곳이 필요하다

씨앗만 살 수 있으면 돈은 다시 돈이 되어 돌아올 뿐이라, 쌓이기만 하고 쓸 데가 없다.
도구 업그레이드가 이 구멍을 메우며, 게임의 성장 축이 된다.

### 상점 위치

마을/맵 전환 없음. 농장 한켠에 칸 두 개를 배치한다.

- **상점 칸** — 씨앗 구매, 도구 업그레이드
- **출하함 칸** — 작물 판매

앞에 서서 스페이스를 누르면 UI가 열린다. 맵 전환이 없어 구현이 가볍고 게임 흐름이 끊기지 않는다.

### 도구 (2종)

괭이(밭 갈기)와 물뿌리개(물 주기). 심기와 수확은 도구 없이 가능.
도구 개수를 늘리기보다 있는 도구를 깊게 만든다.

### 핵심 설계: 체력은 "휘두를 때마다" 소모

체력이 칸당 소모되면 3칸을 한 번에 갈든 세 번에 나눠 갈든 총 체력이 같아, 업그레이드가 키 입력만 줄이고 실익이 없다.

**휘두름 단위로 소모하게 하면 칸당 효율이 올라간다:**

| 도구 | 처리 범위 | 체력 소모 | 칸당 체력 |
|---|---|---|---|
| Lv1 | 바로 앞 1칸 | 2 | 2.0 |
| Lv2 | 바라보는 방향으로 뻗은 일렬 3칸 | 4 | 1.33 |
| Lv3 | 바로 앞 칸을 중심으로 한 3×3칸 | 9 | 1.0 |

괭이와 물뿌리개 모두 동일 구조.

**범위 기준점은 항상 "플레이어가 바라보는 바로 앞 칸"이다.**
Lv2는 그 칸부터 바라보는 방향으로 3칸이 뻗어나가고, Lv3은 그 칸을 한가운데 두는 3×3이다.

**밭 밖으로 나가는 칸은 계산에서 제외한다.** 체력은 실제로 처리된 칸 수와 무관하게 휘두름당 정액으로 소모된다 — 가장자리에서 휘두르면 손해지만, 그 판단은 플레이어의 몫이다.

### 성장 루프

```
돈을 번다 → 도구를 올린다 → 하루에 더 넓은 밭을 돌본다 → 돈을 더 번다
```

돈에 쓸 곳이 생기는 동시에 그 지출이 다시 수익을 만들어 루프가 계속 굴러간다.

### 데이터 모델 영향 (최소)

- `tool_levels`에 도구별 레벨 정수 하나씩만 추가 저장
- 업그레이드 = 돈 차감 + 레벨 증가
- 실제 구현 작업은 **"레벨에 따라 영향받는 칸 목록을 계산하는 함수" 하나**. GameState 내부에 있어 테스트 용이

### 명시적으로 제외한 것

- **업그레이드 소요 시간 없음** — 스타듀밸리는 도구를 맡기면 이틀 못 쓰지만, 그 긴장은 도구가 여러 개일 때 성립한다. 이 규모에서는 짜증만 남는다. 돈을 내면 즉시 적용.

## 4. 숫자

**전제: 이 숫자들은 전부 틀렸을 것이다.**
밸런스는 스프레드시트가 아니라 플레이로 맞춘다. 여기 적힌 값은 "플레이 가능한 출발점"이지 정답이 아니다.

**중요한 것은 숫자 자체가 아니라 한 곳에 모아두는 것.** 상수 파일 하나에 전부 넣어두면 조정이 파일 하나 고치는 일이 된다.

### 시간과 체력

| 항목 | 값 |
|---|---|
| 하루 시작 | 오전 6:00 |
| 하루 강제 종료 | 새벽 2:00 |
| 시계 속도 | 현실 10초 = 게임 10분 (하루 ≈ 실제 20분) |
| 최대 체력 | 60 |

개발 중 테스트를 위해 시계 배속 옵션을 넣어둔다.

### 체력 소진과 기절 규칙

**행동은 체력이 1이라도 남아 있으면 수행된다.** 체력이 모자라도 거부되지 않고, 0까지 깎이면서 수행된 뒤 그 자리에서 기절한다.

- 체력이 0이 되는 순간 → 기절 → 다음 날 아침 6:00, 체력 `max_energy / 2`로 시작
- 새벽 2:00에 도달해도 동일하게 기절 처리
- 침대에서 자면 → 다음 날 아침 6:00, 체력 `max_energy` 전량으로 시작

**"체력이 부족하면 행동을 거부한다"는 규칙은 채택하지 않는다.** 그렇게 하면 체력이 0에 닿기 전에 모든 행동이 막혀 기절이 영영 발생하지 않고, 아무것도 못 하는데 하루도 안 끝나는 어정쩡한 상태가 생긴다.

### 작물 3종

| 작물 | 성장 | 씨앗 | 판매 | 순익/일 |
|---|---|---|---|---|
| 순무 | 3일 | 20 | 45 | 8.3 |
| 감자 | 5일 | 50 | 105 | 11 |
| 호박 | 8일 | 120 | 280 | 20 |

**표에 드러나지 않는 비용이 실제 선택을 만든다:**

- 호박은 8일 내내 물을 줘야 하므로 칸당 체력 16 소모 (순무는 6)
- 하루 체력이 60이므로 "호박으로 도배하면 물 주다 하루가 끝나는" 상황이 발생
- 8일간 자본이 묶이므로 초반에는 구매 자체가 불가능

이것이 의도된 긴장이다. 표만 보면 호박이 무조건 이득으로 보이지만 실제로는 아니다.

### 도구 업그레이드 가격

| | 괭이 | 물뿌리개 |
|---|---|---|
| Lv2 | 500 | 500 |
| Lv3 | 2000 | 2000 |

첫 업그레이드는 며칠 모으면 닿고, Lv3은 장기 목표가 되도록 설정.

### 시작 상태

- 돈 200 (순무 씨앗을 더 살 수 있는 정도 → 첫날부터 상점을 만나게 됨)
- 순무 씨앗 5개
- 모든 도구 Lv1
- 밭 전체 `UNTILLED`

### 밭 크기: 10×8 = 80칸

**Lv1 물뿌리개로는 하루 30칸(체력 60)이 한계라 처음엔 절반도 못 쓴다. 의도된 것이다.**
밭이 남아도는 게 눈에 보여야 도구 업그레이드 동기가 생긴다. 밭이 딱 맞으면 업그레이드할 이유가 사라진다.

## 5. 저장과 테스트

### 저장 시점: 하루가 넘어갈 때 자동 저장

하루 넘김은 게임에서 **유일하게 상태가 확실히 안정된 시점**이다. 물 주다 만 애매한 중간 상태가 아니라, 성장 처리가 끝나고 아침 6:00으로 리셋된 깨끗한 지점.

이 시점에만 저장하면 "저장 중 상태가 반쯤 바뀐" 문제가 구조적으로 발생하지 않는다.

**따라서 불필요한 것:** 수동 저장 버튼, 다중 저장 슬롯.

### 저장 형식

- 경로: `user://save.json` (Windows 실제 경로: `%APPDATA%\Godot\app_userdata\farm-game\`)
- 게임 시작 시 파일이 있으면 불러오고, 없으면 새 게임
- 구현: `FarmModel`에 함수 두 개 — 상태 → Dictionary, Dictionary → 상태
- 파일 읽기/쓰기 자체는 `GameState`(autoload)가 담당

섹션 1에서 상태를 한 곳에 모아둔 설계가 여기서 값을 한다.

### 테스트 도구: 직접 만든 스크립트로 시작

Godot에는 내장 테스트 러너가 없다. 애드온(GUT이 사실상 표준)을 쓰거나 직접 만들어야 한다.

**첫 프로젝트에서는 직접 만드는 쪽을 택한다.** 테스트 스크립트를 `godot --headless --script`로 실행.
애드온을 얹으면 문제가 생겼을 때 "내 코드 문제인지 애드온 문제인지" 구분이 안 되는 상황이 생긴다.
테스트가 많아지면 GUT으로 이전한다.

**이것이 가능한 이유는 섹션 1의 구조 덕분이다.** `FarmModel`은 노드가 아니므로 씬 트리 없이 생성하고 함수를 호출할 수 있다. 테스트는 `GameState`(autoload)를 건드리지 않고 `FarmModel`만 대상으로 한다.

### 필수 테스트 항목

- 물 안 준 작물은 안 자라고, 물 준 작물은 자란다
- 수확 시 인벤토리가 증가하고 해당 칸이 `TILLED`로 복귀한다
- 체력이 0이 되면 기절하고, 다음 날 체력 절반으로 시작한다
- 침대에서 자면 체력이 전량 회복된다
- 저장 → 불러오기 라운드트립 후 상태가 동일하다
- **도구 레벨별 영향 칸 계산 — 특히 밭 가장자리**

마지막 항목이 특히 중요하다. 밭 끝에서 Lv3 괭이(3×3)를 사용하면 밭 밖의 칸을 참조하게 되며, 여기서 문제가 생길 가능성이 높다. 밭 밖 칸이 조용히 제외되는지, 체력은 그대로 정액 소모되는지를 테스트로 못박는다.
