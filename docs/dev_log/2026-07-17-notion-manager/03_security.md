## Critical (must fix before implementation)

- **[C1] 로그 시크릿 마스킹 필터가 "루트 로거에 부착"되는 지점이 불명확 — 실제 유출 경로(discord.py/SDK 로그)를 놓칠 수 있음**
  스펙 §4.1: "`SecretMaskingFilter`... 루트 로거에 부착". Python `logging`의 propagation 모델에서, 필터를 **Logger 객체**에 붙이면(`logging.getLogger().addFilter(f)`) 그 필터는 해당 로거로 **직접** `.debug()/.info()` 등을 호출한 레코드에만 적용되고, `discord`, `openai`, `anthropic`, `httpx`, `notion_client` 등 하위 named logger가 만든 레코드가 상위(root)로 전파(propagate)될 때는 재검사되지 않는다(전파 시 검사되는 것은 **Handler**의 필터뿐). 즉 "루트 로거에 부착"을 문자 그대로 구현하면, 리뷰 브리프가 정확히 지목한 유출 경로 — discord.py 자체 HTTP/게이트웨이 디버그 로그, openai/anthropic SDK의 내부 로거 — 는 마스킹을 **우회**할 수 있다. 이는 불변 제약 1(시크릿 비노출) 위반 가능성.
  — 근거: §4.1, 계획 파일 `notion-manager/config.py` (`SecretMaskingFilter`, `setup_logging`).
  — 해결안: 필터를 root 로거가 아니라 root의 **Handler(들)**에 부착하거나(`handler.addFilter(...)`), 사용되는 모든 3rd-party 로거(`discord`, `discord.http`, `openai`, `anthropic`, `httpx`, `notion_client`)에 명시적으로 동일 필터를 부착하도록 스펙에 못박는다. `tests/test_config.py`에 "하위 named logger를 통해 로깅해도 마스킹된다"는 케이스를 추가해 회귀를 막는다.

- **[C2] update 툴이 page_id의 소속 데이터소스를 검증하지 않음 — "update로 위장한" 워크스페이스 구조 변경 가능**
  §2.3 디스패처 검증 목록(미지 툴/인자, enum 밖 값, 날짜/범위, archived 거부)에 **page_id가 실제로 ds_schedule/ds_tracker 소속 행인지 확인하는 절차가 없다**. 그런데 §5.6-5에 따라 `workspace_map.json` **전체**(멤버 5인 페이지 ID, 스터디 가이드 페이지 ID, 루트 페이지 ID 포함)가 시스템 프롬프트에 원문 그대로 주입된다. Notion API의 `pages.update`는 데이터베이스 행이 아닌 일반 페이지에도 공통으로 존재하는 `title` 속성을 그대로 받아들이므로, 모델이 (프롬프트 인젝션이든 환각이든) `update_schedule_entry(page_id="<멤버 페이지 ID>", title="...")` 류를 호출하면 실제로 **멤버 페이지/가이드 페이지/루트 페이지의 제목이 변경**될 수 있다 — §2.3 표에서 요구하는 "update로 위장한 파괴적 변경(제목/속성 전부 덮어쓰기)" 우회 경로가 실제로 존재한다.
  추가 근거: §4.1 필수 키 검증 목록에 `NOTION_ROOT_PAGE_ID`가 없다(즉 config.py가 이 값의 존재를 강제하지 않는다) — 이는 `.env.example`상 integration이 루트 페이지 단위로 캐스케이드 공유되어 있을 가능성을 시사하며, 그렇다면 위 update 호출이 Notion 측 권한 검사에서도 막히지 않고 그대로 성공한다. 이는 불변 제약 2("Notion 워크스페이스 구조를 임의로 변경하지 않는다")의 직접 위반 경로.
  — 근거: §2.3, §2.5, §4.4(L2 표), §5.6-5, 계획 파일 `notion-manager/agent/notion_api.py`(`NotionGateway.update_page`), `notion-manager/agent/tools.py`(`ToolDispatcher`).
  — 해결안: `update_page` 호출 전 `pages.retrieve(page_id)`로 `parent.data_source_id`가 툴이 기대하는 ds_schedule/ds_tracker와 일치하는지 검증(불일치 시 ok=false 거부), 또는 세션 내 이번 요청의 `query_*`/`create_*` 결과로 실제 관측된 page_id만 화이트리스트로 허용. 두 방법 중 하나를 §2.3/§4.4에 명문화할 것.

- **[C3] 쓰기 3회 상한이 "요청(메시지 1건)" 단위로 리셋 — 대량 변경 차단의 실질 보장이 없음, §9 리스크3의 안전 주장과 모순**
  §4.4 L2: "요청(사용자 메시지 1건)당 쓰기 툴 호출 상한 3회". 스펙 자신도 이 한도가 "되묻기→재요청으로 우회 가능"함을 인지하고 이를 "정상 사용 저해 없음"이라는 이유로 **의도적으로 허용**한다. 그러나 트리거 C(§2.1)에 의해 같은 스레드의 후속 메시지는 멘션 없이도 전부 같은 세션으로 처리되므로, 사용자가 "3개만 바꿔줘"를 연달아 여러 번 보내면 한 세션 안에서 **사실상 무제한 쓰기**가 가능하다. 이는 리뷰 브리프가 지목한 정확한 우회 경로("같은 스레드 연속 메시지로 나눠 시키기")이며, 동시에 **§9 리스크3의 "쓰기 상한 3회 — 최악 피해는 '잘못된 항목 3건'으로 상한"이라는 주장과 정면으로 모순**된다(실제로는 3건이 아니라 세션이 유지되는 한 무제한). 불변 제약 4("삭제/대량 변경: MVP에서는 미지원 응답 처리")가 명시적 "삭제해줘" 문구에는 L1으로 막히지만, 여러 정상적 요청을 잘게 쪼갠 사실상의 대량 변경에는 아무 정량적 상한이 없다.
  — 근거: §4.4(L2), §9 리스크3, §2.1 트리거 C, 계획 파일 `notion-manager/agent/tools.py`(`ToolDispatcher.write_count`).
  — 해결안: (a) 세션(스레드) 단위 누적 카운터(예: 스레드 생성 후 N분 또는 세션 전체에 대해 쓰기 M회) 추가, 또는 (b) §9의 안전 주장을 "메시지당 3건, 스레드 내 재요청 횟수에는 상한 없음"으로 정정하고 이 잔여 리스크를 CTO가 명시적으로 승인하도록 게이트에 올릴 것. 최소한 스펙 문서 내부의 모순(§4.4 vs §9-3)은 반드시 해소되어야 함.

- **[I1] 발화자→멤버 추정이 스푸핑 가능한 Discord 표시명(닉네임)에 의존 — 신원 오귀속/사칭**

- **[I3] DM 판별 순서 미명시 — `message.guild`가 None일 때 예외 발생 가능**

## Minor (nice to have)

- **[M1] SQLite 세션 DB의 파일 권한/암호화 미언급**
  §2.2: 대화 원문 전체(사용자 메시지, tool 인자/결과)가 평문으로 `sessions.db`에 저장된다. 5인 소규모 로컬 운영 전제라 위험도는 낮으나, 다중 사용자가 접근 가능한 호스트에 배포될 경우 대화 내용이 그대로 노출될 수 있다. §9 리스크8과 유사한 톤으로 README에 "이 DB 파일을 공유 서버/다중 사용자 호스트에 두지 말 것" 안내를 추가할 것.

- **[M2] Notion integration 토큰의 실제 공유 범위(페이지 스코프) 미확인**
  C2가 해결되어도, Notion 통합이 실제로 어느 페이지들까지 연결(공유)되어 있는지(루트 페이지 캐스케이드 공유 vs 개별 DB 3개만)를 스펙이 확인하지 않았다. §4.1 필수 키 목록에 `NOTION_ROOT_PAGE_ID`가 빠진 것도 이 의문을 뒷받침한다. 애플리케이션 코드 검증(C2)이 유일한 방어선이 되지 않도록, Notion 측 권한 범위를 좁히는 것이 방어 심층화(defense in depth)에 해당한다.
  — 해결안: 구현/배포 체크리스트에 "Notion integration이 실제로 연결된 페이지 목록 확인, 불필요하게 넓으면(예: 루트 캐스케이드) 3개 DB로 축소"를 항목으로 추가.

- **[M3] 사용자에게 노출되는 에러 요약(E2)이 로그 마스킹과 같은 가공을 거치는지 불명**
  §5.5 E2: "노션에 반영하지 못했어요. 원인: {한국어 요약}". 이 "한국어 요약"이 §4.1의 로그 마스킹 규칙(예외 클래스명+상태코드 요약, repr 금지)과 **동일한 가공 함수**를 거치는지, 아니면 별도 경로인지 §2.5/§4.1에 명시되어 있지 않다. Notion 에러 메시지에 시크릿이 담길 가능성은 낮지만, "사용자에게 보여주는 에러 문자열도 로그와 동일한 마스킹/요약 함수를 거친다"를 명문화하면 이중 방어가 된다.
